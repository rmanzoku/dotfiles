# 共通ルール

- 日本語で応答すること
- MCP サーバーよりも CLI ツールの利用を優先して検討すること
- 人間からの継続的な指示は Memory に依存せず、git 管理されたファイルに保存すること
- 継続的な指示の保存先は、作業リポジトリの `docs/`、`.agents/`、またはグローバル dotfiles（例: `~/.codex/AGENTS.md`）を使うこと
- worktree の削除や新規セッションで失われる前提で扱う情報は Memory ではなく、再現可能な git ファイルとして残すこと
- 一時ファイル、下書き、調査メモなどの一過性ファイルは、作業 worktree 内の `.context/` に生成すること
- 一時ファイルの置き場として `/private` や `/tmp` は使わないこと。worktree 削除後もファイルが残り、再現性と後始末が崩れるため
- アーキテクチャ、運用方針、永続設定、複数ファイルにまたがるワークフロー変更などの大きめの変更では、作業リポジトリの `docs/adr/` に ADR を作成または更新すること
- ADR には、少なくとも作業日時と作業した Agent のモデル名を記載すること

# Plan 共通ルール

## Plan 提示前レビュー（必須）

- Plan をユーザーに提示する前に、必ず簡易レビューを実行すること
- この手順はすべての Plan 提示に適用し、同一セッションの 2 回目以降でも省略してはならない
- 簡易レビューでは、下記 `## レビュー実行` に記載した確認観点を最低限すべて含めること
- レビュー対象は Plan 本文をインライン展開せず、Plan ファイルパスと参考ファイルパスを渡して読ませること
- プロンプトは heredoc (`<<'PROMPT'`) で作成し、引数位置で直接展開せず、いったん変数に格納してから渡すこと
- 実行時は必ず `-C <project_root>` を付け、リポジトリ内ファイルを参照できる状態で実行すること
- モデルと reasoning は明示されたコマンドパターンから変更してはならない
- レビュー結果に `ISSUE` がある場合、Plan を修正してから再レビューすること
- レビュー結果に `ISSUE` があるが、自分で妥当性を判断しきれない場合は、Plan を提示せずユーザーに判断を仰ぐこと
- 次の条件のいずれかに該当する場合、Plan をユーザーへ提示してはならない
  - 簡易レビュー未実行
  - レビュー結果未確認
  - `ISSUE` 未反映
  - `ISSUE` 反映後の再レビュー未実行
  - `ISSUE` の妥当性を判断できないまま未確認

## レビュー実行

- 勝手にモデルとReasoningを変更しないこと。コマンドパターン:

```bash
prompt="$(cat <<'PROMPT'
あなたはプランレビュアーです。以下のプランを簡易レビューしてください。

1. レビュー対象プラン: <plan_file_path>
2. 参考ファイル: <reference_file_paths...>

確認観点:
- ユーザー要求から外れていないか
- スキルを適切に使用しているか
- 実装者に追加判断を残していないか
- 根拠のない決定や先送りが入っていないか
- スコープ外として扱う項目を、理由なく無視していないか
- スコープ外の変更や過剰設計が入っていないか
- 必要なテスト観点と受け入れ条件が欠けていないか

各観点について PASS / ISSUE / SUGGESTION で評価し、ISSUE があれば簡潔な修正案を述べてください。
PROMPT
)"

codex exec --full-auto -m gpt-5.3-codex-spark \
  -c model_reasoning_effort="medium" \
  -C <project_root> \
  "$prompt"
```

# Codex 固有ルール

## Config / Trust 運用

- Codex の trust 設定は原則 `~/.codex/config.toml` に集約し、`[projects."<path>"]` は常用しないこと
- project-scoped `.codex/config.toml` を読む必要がある repo / worktree だけ、例外として対応する `trust_level = "trusted"` を追加すること
- 例外追加前に、その repo に project 固有の Codex 設定が本当に必要か確認すること
- 通常運用で権限不足が出た場合は、恒久設定を緩めず、その実行時だけ `danger-full-access` を使うこと
