# ADR 0010: AI 間の受け渡しは実ファイルを標準とし単発処理のみパイプを許容する

## Context

AI の CLI やサブエージェント機能のあいだで、複数行の指示、構造化データ、長文のプロンプトを受け渡す場面が増えている。
このとき、`-p` などの引数に本文をインライン展開したり、here-doc を直接埋め込んだりすると、
shell quoting の破綻、改行や backtick の崩れ、引数長制限、ログ可読性の低下、子プロセスや別 Agent への再受け渡し失敗が起きやすい。

既存 ADR 0005 では Plan レビューに限って `.context/` の実ファイル経由を必須にしており、
このリポジトリ全体でも一過性ファイルは作業 worktree 内の `.context/` に置く共通ルールがある。
一方で、単一コマンドが標準入力をただちに 1 回だけ読むような単発処理では、パイプは簡潔で安定した手段になりうる。

## Decision

- AI 間や CLI 間で複数行や構造化された内容を受け渡すときは、`.context/` に置いた実ファイル経由を標準とする。
- `-p` などの引数へのインライン展開、here-doc 直書き、fd 依存の受け渡しは原則採用しない。
- パイプでの受け渡しは、単一のコマンドが標準入力をただちに 1 回だけ読む単発処理に限って許容する。
- 再読、再送、別 Agent への移譲、サブプロセス引き継ぎ、監査や再実行のための保存が必要な場合は、パイプではなく実ファイルを使う。
- この判断を repo の `AGENTS.md` と AI 別指示ファイルの共通ルールへ反映する。

## Consequences

- 長文プロンプトや構造化データの受け渡しが shell quoting や引数長制限に左右されにくくなる。
- `.context/` 配下に実体が残るため、失敗時の再実行、比較、監査がしやすくなる。
- パイプを全面禁止しないため、単発の stdin 消費には簡潔さを維持できる。
- inline 展開の自由度は下がるが、多段の AI 連携では再現性と可搬性を優先できる。
